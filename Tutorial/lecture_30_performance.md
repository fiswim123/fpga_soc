# 第三十讲：性能分析与优化 — 从瓶颈到加速

---

## 课程目标

本讲从理论峰值算力出发，深入分析NPU实际推理中的性能瓶颈来源（数据搬运、非计算层、气泡），量化每个瓶颈的影响，并提出优化方向。最后与业界NPU（Google TPU、华为Ascend）进行对比，总结设计得失与未来展望。

**学完本讲你将掌握：**

1. TOPS、周期数、利用率等性能指标的计算方法
2. 性能瓶颈的三大来源及量化分析
3. 总线带宽利用率的测量与优化
4. 与业界NPU的架构对比分析
5. 从瓶颈到加速的系统性优化思路

### 1.2 理论峰值算力计算

**源码位置：** `docs/DESIGN_REPORT.md` 第4.3.1节

```
TOPS_peak = (MAC_count × 2 × f) / 10^12

其中:
  MAC_count = 总MAC单元数
  2         = 每MAC每周期2次运算 (1次乘法 + 1次加法)
  f         = 工作频率 (Hz)
  10^12     = Tera (万亿)
```

**本设计参数：**

```
┌──────────────────────────────────────────────────┐
│  本设计NPU配置                                    │
│                                                  │
│  计算核数量:     8                                │
│  每核PE数量:     16 × 16 = 256                   │
│  总MAC数量:      8 × 256 = 2048                  │
│  每MAC运算数:    2 (INT8乘 + INT32累加)           │
│  工作频率:       200 MHz                          │
│                                                  │
│  TOPS_peak = 2048 × 2 × 200×10^6 / 10^12        │
│           = 819,200 × 10^6 / 10^12               │
│           = 0.8192 TOPS                           │
│                                                  │
│  满足赛题要求 ≥ 0.5 TOPS                          │
│  接近优化目标 ≈ 1 TOPS                            │
└──────────────────────────────────────────────────┘
```

**不同配置的理论算力：**

| 配置 | MAC数量 | 峰值算力@200MHz | 相对于8核 |
|------|---------|-----------------|-----------|
| 1核×16×16 | 256 | 0.102 TOPS | 12.5% |
| 2核×16×16 | 512 | 0.205 TOPS | 25% |
| 4核×16×16 | 1024 | 0.410 TOPS | 50% |
| 8核×16×16（当前）| 2048 | 0.819 TOPS | 100% |
| 12核×16×16 | 3072 | 1.229 TOPS | 150% |
| 16核×16×16 | 4096 | 1.638 TOPS | 200% |

### 设计视角：为什么这样设计？

性能分析的核心目标是理解实际性能与理论峰值之间的差距, 找到瓶颈并指导优化方向。

**核心问题 1：为什么利用率只有 ~50-83%？**

```
  利用率损失的三个来源:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  理论峰值: 0.819 TOPS (2048 MAC × 2 × 200MHz)          │
  │  实际实测: 0.678 TOPS (ResNet-18)                       │
  │  利用率:   82.7%                                        │
  │  损失:     17.3%                                        │
  │                                                         │
  │  损失分解:                                              │
  │                                                         │
  │  ┌─────────────────────────────────────────┐            │
  │  │ 瓶颈 1: 脉动阵列填充/排空 (4.7%)        │            │
  │  │                                         │            │
  │  │ 原因: 脉动阵列的固有特性                 │            │
  │  │ · Fill 阶段: 数据逐步流入, 部分 PE 空闲  │            │
  │  │ · Drain 阶段: 数据逐步流出, 部分 PE 空闲 │            │
  │  │ · 16×16 阵列: Fill+Drain = 62 周期/次   │            │
  │  │ · 大矩阵时开销比低, 小矩阵时开销比高    │            │
  │  └─────────────────────────────────────────┘            │
  │                                                         │
  │  ┌─────────────────────────────────────────┐            │
  │  │ 瓶颈 2: DMA 数据搬运延迟 (6.2%)         │            │
  │  │                                         │            │
  │  │ 原因: DMA 搬运与计算串行执行             │            │
  │  │ · 搬运期间 MAC 单元空闲                  │            │
  │  │ · Layer 2 DMA: 5600 周期 (占总 32%)     │            │
  │  │ · 通过流水线重叠部分隐藏                 │            │
  │  │ · 优化方向: 双缓冲 (完全重叠)            │            │
  │  └─────────────────────────────────────────┘            │
  │                                                         │
  │  ┌─────────────────────────────────────────┐            │
  │  │ 瓶颈 3: 层间流水线气泡 (6.4%)            │            │
  │  │                                         │            │
  │  │ 原因: 层间切换的额外开销                 │            │
  │  │ · CSR 配置: ~30-40 周期/层              │            │
  │  │ · 流水线排空+重填: ~62 周期/层           │            │
  │  │ · 存储切换: ~5-10 周期/层               │            │
  │  │ · 对深层网络影响更大 (层数 × 开销)       │            │
  │  └─────────────────────────────────────────┘            │
  │                                                         │
  └─────────────────────────────────────────────────────────┘

  为什么不能达到 100%?
  · 脉动阵列的 Fill/Drain 是固有开销, 无法完全消除
  · DMA 与计算的串行关系需要硬件支持才能重叠
  · 层间切换需要配置时间, 除非实现全自动层序列器
  · 82.7% 的利用率在业界属于良好水平 (TPU v1 ~85%)
```

**核心问题 2：瓶颈在哪里？如何定位？**

```
  瓶颈定位方法:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  方法 1: 理论分析 (本设计使用)                          │
  │  · 分解每层的计算周期、DMA 周期、配置周期               │
  │  · 计算各部分占比, 找到最大占比项                       │
  │  · 优点: 快速, 不需要仿真                               │
  │  · 缺点: 可能遗漏实际实现中的额外开销                   │
  │                                                         │
  │  方法 2: 性能计数器 (Performance Counters)               │
  │  · 在硬件中添加周期计数器                               │
  │  · 统计各阶段的实际周期数                               │
  │  · 优点: 实际数据, 精确                                 │
  │  · 缺点: 需要额外硬件面积                               │
  │                                                         │
  │  方法 3: 仿真波形分析                                   │
  │  · 在仿真中观察信号波形                                 │
  │  · 统计 MAC 有效计算周期 vs 空闲周期                    │
  │  · 优点: 最精确                                         │
  │  · 缺点: 耗时, 需要人工分析                             │
  │                                                         │
  │  本项目使用方法 1 + 方法 3:                             │
  │  · 理论分析确定大方向                                   │
  │  · 仿真波形验证具体数值                                 │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**核心问题 3：为什么总线利用率 86.7% 而不是更高？**

```
  总线利用率的理论上限:

  η = L / (1 + L) × 100%

  其中 L = AXI Burst 长度 (beat 数)

  ┌──────────┬──────────────┬─────────────────────────────┐
  │ Burst长度 │ 理论利用率    │ 限制因素                     │
  ├──────────┼──────────────┼─────────────────────────────┤
  │ 1        │ 50.0%        │ 每次传输 1 拍, 地址握手开销  │
  │ 4        │ 80.0%        │ CPU 典型 burst 长度          │
  │ 8        │ 88.9%        │ 中等 burst                   │
  │ 16       │ 94.1%        │ 本设计 DMA 最大 burst        │
  │ 256      │ 99.6%        │ 接近 100%, 但需要大缓冲      │
  └──────────┴──────────────┴─────────────────────────────┘

  本设计选择 burst=16 的权衡:
  · burst 越大 → 利用率越高, 但需要更大的 FIFO 缓冲
  · burst=16 → 利用率 94.1%, FIFO 深度 16 即可
  · burst=256 → 利用率 99.6%, 但 FIFO 需要 256 深度 → 面积大增
  · burst=16 是面积与利用率的最佳平衡点

  实测 86.7% < 理论 94.1% 的原因:
  · CPU 的短 burst (L=4) 拉低了加权平均
  · 总线仲裁开销 (round-robin 需要额外周期)
  · 地址握手延迟 (arready/awready 等待)
```

### 设计视角：如何从零开始设计？

分析和优化 SoC 性能需要系统化的方法。以下是五步设计流程。

**步骤 1：建立性能模型**

```
  性能模型的输入和输出:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  输入:                                                  │
  │  · 网络结构 (每层的类型、参数、尺寸)                    │
  │  · 硬件参数 (MAC 数量、频率、burst 长度、存储容量)      │
  │  · 数据流策略 (权重静止/输出静止/行静止)                │
  │                                                         │
  │  输出:                                                  │
  │  · 每层推理周期数                                       │
  │  · 总推理周期数                                         │
  │  · 理论峰值算力                                         │
  │  · 实际算力                                             │
  │  · 利用率                                               │
  │  · 各瓶颈占比                                           │
  │                                                         │
  │  模型精度:                                              │
  │  · 理论模型: 精度 ±5% (忽略实现细节)                   │
  │  · 仿真验证: 精度 ±1% (考虑实际时序)                    │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**步骤 2：分解推理时序**

```
  每层推理时序分解模板:

  Layer N 推理周期 = DMAC 填充 + MAC 计算 + 层间开销

  DMAC 填充:
  · im2col 变换行数 = ceil(IC × KH × KW / TILE_ROWS) × OH × OW
  · 每行 1 周期写入 SA_RAM
  · 填充周期 = 行数 × 1

  MAC 计算:
  · tile 数 = ceil(OC / TILE_COLS)
  · 每 tile: 读取列数 + drain + 写回行数
  · 计算周期 = tile 数 × (列数 + drain + 行数)

  层间开销:
  · CSR 配置: ~30-40 周期
  · 流水线排空+重填: ~62 周期
  · 存储切换: ~5-10 周期
  · 总计: ~100 周期/层

  示例 (Conv1: 5×5, 3→32, 32×32):
  · DMAC: ceil(75/40) × 1024 = 2 × 1024 = 2048 周期
  · MAC: 2 tiles × (75 + 8 + 40) = 246 周期
  · 层间: 100 周期
  · 总计: 2394 周期
```

**步骤 3：识别和量化瓶颈**

```
  瓶颈量化分析表:

  ┌──────────────┬──────────┬──────────┬──────────┬──────────┐
  │ 瓶颈          │ 周期数    │ 占比      │ 可优化?  │ 优化方案  │
  ├──────────────┼──────────┼──────────┼──────────┼──────────┤
  │ DMAC 填充     │ 7,550    │ 33.2%    │ 部分     │ 流式im2col│
  │ MAC 计算      │ 15,165   │ 66.8%    │ 部分     │ 更大阵列  │
  │ Fill/Drain    │ ~1,071   │ 4.7%     │ 困难     │ 固有开销  │
  │ DMA 搬运      │ ~1,408   │ 6.2%     │ 可以     │ 双缓冲    │
  │ 层间气泡      │ ~1,453   │ 6.4%     │ 可以     │ 自动序列器│
  ├──────────────┼──────────┼──────────┼──────────┼──────────┤
  │ 总计          │ 22,715   │ 100%     │          │          │
  └──────────────┴──────────┴──────────┴──────────┴──────────┘

  瓶颈优先级排序 (按优化收益):
  1. DMA 搬运: 双缓冲可完全消除 → 节省 6.2%
  2. 层间气泡: 自动序列器可大幅减少 → 节省 ~4%
  3. DMAC 填充: 流式 im2col 已实现, 进一步优化有限
  4. Fill/Drain: 固有开销, 无法消除
```

**步骤 4：设计优化方案**

```
  优化方案设计框架:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  优化 1: 双缓冲 (Double Buffering)                      │
  │  · 目标: 消除 DMA 搬运延迟                              │
  │  · 原理: DMA 写 Buffer_B 的同时, MAC 计算 Buffer_A      │
  │  · 代价: 存储空间翻倍 (256KB → 512KB)                   │
  │  · 收益: ~6.2% 利用率提升                               │
  │  · 复杂度: 中 (需要缓冲区管理 FSM)                      │
  │                                                         │
  │  优化 2: 硬件层序列器                                   │
  │  · 目标: 减少层间配置开销                               │
  │  · 原理: 预存每层配置, 自动切换                         │
  │  · 代价: 额外控制逻辑 + 配置存储                        │
  │  · 收益: ~4% 利用率提升                                 │
  │  · 复杂度: 高 (需要设计自动配置 FSM)                    │
  │                                                         │
  │  优化 3: 更大阵列 (16×16 → 32×32)                      │
  │  · 目标: 提升稳态利用率                                 │
  │  · 原理: 更大阵列的 Fill/Drain 开销比更低               │
  │  · 代价: MAC 数量 4×, 面积 ~4×                          │
  │  · 收益: 大矩阵利用率提升, 小矩阵可能下降               │
  │  · 复杂度: 低 (复制现有设计)                            │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**步骤 5：验证优化效果**

```
  优化验证流程:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  Step 1: 更新性能模型                                   │
  │  · 将优化方案的参数代入模型                             │

  │                                                         │
  │  Step 2: RTL 实现优化                                   │

  │                                                         │
  │  Step 3: 仿真验证                                       │
  │  · 运行功能测试 (确保优化不破坏正确性)                  │
  │  · 收集性能数据 (周期数、利用率)                        │
  │                                                         │
  │  Step 4: 对比分析                                       │

### 设计视角：架构模式与原则

性能分析和优化中有多种可复用的模式。掌握这些可以系统化地提升 SoC 性能。

**模式 1：性能建模模式（Performance Modeling Pattern）**

```
  核心思想: 在实现之前用数学模型预测性能, 指导设计决策

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  性能建模的三个层次:                                    │
  │                                                         │
  │  Level 1: 粗粒度模型 (设计初期)                         │
  │  · 输入: MAC 数量、频率、网络总 MAC 数                  │
  │  · 输出: 理论推理时间                                   │
  │  · 精度: ±30%                                          │
  │  · 用途: 架构探索 (选多少核? 多大阵列?)                 │
  │                                                         │
  │  Level 2: 中粒度模型 (架构确定后)                       │
  │  · 输入: 每层参数、tile 策略、burst 长度               │
  │  · 输出: 每层周期数、总周期数、各瓶颈占比               │
  │  · 精度: ±10%                                          │
  │  · 用途: 瓶颈分析、优化方向确定                         │
  │                                                         │
  │  Level 3: 细粒度模型 (RTL 完成后)                       │
  │  · 输入: 仿真波形、实际周期计数                         │
  │  · 输出: 精确的每层周期数、利用率                       │
  │  · 精度: ±1%                                           │
  │  · 用途: 最终性能确认、优化效果验证                     │
  │                                                         │
  │  建模原则:                                              │
  │  · 从粗到细, 逐步精确                                   │
  │  · 每个层次都要与实测数据校准                           │
  │  · 模型参数随设计迭代更新                               │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**模式 2：瓶颈分析模式（Bottleneck Analysis Pattern）**

```
  核心思想: 找到性能链中最薄弱的环节, 集中优化资源

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  瓶颈分析流程:                                          │
  │                                                         │
  │  Step 1: 绘制性能链                                     │
  │                                                         │
  │  DMA搬运 ──► im2col ──► MAC计算 ──► 重量化 ──► 结果写回 │
  │  (6.2%)     (组合逻辑)  (66.8%)    (重叠)     (重叠)   │
  │                                                         │
  │  Step 2: 找到最长环节 (瓶颈)                            │
  │  · MAC 计算占 66.8% → 主要瓶颈                         │
  │  · 但 MAC 计算本身无法大幅优化 (已是 100% 利用)         │
  │  · 真正可优化的是 DMA 搬运 (6.2%) 和层间气泡 (6.4%)    │
  │                                                         │
  │  Step 3: 优化瓶颈                                       │
  │  · 双缓冲: 将 DMA 搬运与 MAC 计算重叠                  │
  │  · 优化后: DMA 搬运从瓶颈中消除                         │
  │  · 新瓶颈: 层间气泡 (6.4%) 成为主要可优化项            │
  │                                                         │
  │  Step 4: 重复 Step 2-3                                  │
  │  · 直到所有可优化瓶颈都被处理                           │
  │  · 或收益递减到不值得继续                               │
  │                                                         │
  │  瓶颈分析的直觉:                                        │
  │  · Amdahl 定律: 优化占比 10% 的瓶颈, 最多提升 11%     │
  │  · 优先优化占比最大的可优化瓶颈                         │
  │  · 固有开销 (Fill/Drain) 不要浪费精力去优化             │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**模式 3：数据流优化模式（Dataflow Optimization Pattern）**

```
  核心思想: 通过重叠计算和数据搬运来隐藏延迟

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  数据流优化的三个层次:                                  │
  │                                                         │
  │  Level 1: 流式处理 (Streaming)                          │
  │  · 原理: 数据生成一行, 立即消费一行                     │
  │  · 示例: 流式 im2col (组合逻辑地址计算, 0 周期延迟)    │
  │  · 效果: 消除 im2col 的存储写入延迟                     │
  │  · 本设计: 已实现                                       │
  │                                                         │
  │  Level 2: 双缓冲 (Double Buffering)                     │
  │  · 原理: DMA 写 Buffer_B 的同时 MAC 读 Buffer_A        │
  │  · 时序:                                                │
  │    传统: DMA_L1 → MAC_L1 → DMA_L2 → MAC_L2            │
  │    双缓冲: DMA_L1 → DMA_L2 → DMA_L3 ...               │
  │              MAC_L1 → MAC_L2 → MAC_L3 ...              │
  │    (搬运与计算完全重叠)                                  │
  │  · 效果: 消除 DMA 搬运延迟                              │
  │  · 本设计: 未实现 (优化方向)                             │
  │                                                         │
  │  Level 3: 权重预取 (Weight Prefetching)                 │
  │  · 原理: 边计算边加载下一 tile 的权重                   │
  │  · 需要: 双端口权重 Buffer 或双 Bank                    │
  │  · 效果: 减少权重加载开销 ~30%                          │
  │  · 本设计: 未实现 (优化方向)                             │
  │                                                         │
  │  数据流优化的收益递减规律:                               │
  │  · Level 1 (流式): 收益大, 实现简单                     │
  │  · Level 2 (双缓冲): 收益中, 实现中等                   │
  │  · Level 3 (预取): 收益小, 实现复杂                     │
  │                                                         │
  └─────────────────────────────────────────────────────────┘

  双缓冲控制器 FSM 示例:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  IDLE ──► LOAD_A ──► COMPUTE_A ──► LOAD_B ──► COMPUTE_B │
  │   ▲        │ DMA写A    │ MAC读A     │ DMA写B    │ MAC读B│
  │   │        │           │ 同时DMA写B  │           │ 同时  │
  │   │        │           │            │           │ DMA写A│
  │   └────────┴───────────┴────────────┴───────────┘       │
  │                                                         │
  │  关键: LOAD_B 与 COMPUTE_A 并行执行                     │
  │  条件: Buffer A 和 Buffer B 独立, 无数据依赖            │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

## 2. 实际推理性能分析

### 2.1 完整推理时序分解

**源码位置：** `docs/NPU_DESIGN_REPORT.md` 第7节

以CIFAR-10 TinyCIFAR10_5x5网络为例，分解每一层的推理周期：

```
网络结构:
  Conv1: 5×5, 3→32ch, pad=2  → 32×32×32 → ReLU → MaxPool → 16×16×32
  Conv2: 5×5, 32→64ch, pad=2 → 16×16×64 → ReLU → MaxPool → 8×8×64
  GAP:   8×8×64 → 64
  FC:    64→10
```

**Layer 1 推理时序：**

```
Phase 1: DMAC填充 image_sa_ram (im2col变换)
  活动行数 = ceil(1024/40) × 75 = 26 × 75 = 1950 行
  每行1周期写入
  耗时: ~1950 周期

Phase 2: MAC计算 (26个tile)
  每个tile:
    ┌────────────────────────────────────────┐
    │ 从image_sa_ram读取75列 → 75 周期       │
    │ 脉动阵列计算(含drain)   → 8 周期        │
    │ 结果写回result_ram(40行) → 40 周期      │
    │ PPU并行处理 (与写回重叠) → 0 额外周期    │
    ├────────────────────────────────────────┤
    │ 每tile小计: 75 + 8 + 40 = 123 周期      │
    └────────────────────────────────────────┘
  26 tiles × 123 cycles/tile = 3198 周期

Layer 1 总计: 1950 + 3198 ≈ 5148 周期
```

**Layer 2 推理时序：**

```
Phase 1: DMAC填充 image_sa_ram
  活动行数 = ceil(256/40) × 800 = 7 × 800 = 5600 行
  耗时: ~5600 周期

Phase 2: MAC计算 (2 Pass × 7 Tile)
  每tile:
    读取800列 → 800 周期
    drain      → 8 周期
    写回40行   → 40 周期
    小计: 848 周期

  2 Pass × 7 Tile × 848 = 11872 周期

Layer 2 总计: 5600 + 11872 ≈ 17472 周期
```

**FC层推理时序：**

```
GAP: 0 额外周期 (与卷积阶段并行累积)
FC:  10类 × 8周期/类 + 状态机开销 ≈ 95 周期
```

**总推理周期：**

```
┌────────────┬──────────┬───────────┬───────────┐
│ 层          │ DMAC填充 │ MAC计算    │ 总周期     │
├────────────┼──────────┼───────────┼───────────┤
│ Conv1+Pool  │ 1,950    │ 3,198     │ 5,148     │
│ Conv2+Pool  │ 5,600    │ 11,872    │ 17,472    │
│ GAP+FC      │ 0        │ 95        │ 95        │
├────────────┼──────────┼───────────┼───────────┤
│ 总计        │ 7,550    │ 15,165    │ 22,715    │
└────────────┴──────────┴───────────┴───────────┘

@100MHz → ~227 μs
@200MHz → ~114 μs
```

## 3. 性能瓶颈分析

### 3.1 瓶颈来源分类

```
理论峰值: 0.819 TOPS (2048 MAC × 2 × 200MHz)

实际实测: 0.678 TOPS (ResNet-18)

差距: 0.819 - 0.678 = 0.141 TOPS (17.2%损失)

损失来源分解:
┌─────────────────────────────────────────────────┐
│                                                 │
│  ┌───────────────────┐  4.7%                   │
│  │ 脉动阵列填充/排空   │  (R+C-1周期开销)        │
│  └───────────────────┘                         │
│  ┌───────────────────┐  6.2%                   │
│  │ DMA数据搬运延迟     │  (权重/特征图加载)       │
│  └───────────────────┘                         │
│  ┌───────────────────┐  6.4%                   │
│  │ 层间流水线气泡      │  (配置/同步等待)         │
│  └───────────────────┘                         │
│                                                 │
│  总损失: ~17.3%                                 │
│  实际利用率: ~82.7%                             │
└─────────────────────────────────────────────────┘
```

### 3.2 瓶颈一：脉动阵列填充与排空

```
脉动阵列的固有开销:

  对于 R行×C列 的阵列, 计算 M×K × K×N 的矩阵乘法:

  ┌─────────────────────────────────────────────┐
  │                                             │
  │  填充阶段 (Fill):  R+C-1 周期               │
  │  │ 数据从左上角逐步流入阵列                   │
  │  │ 部分PE尚未开始有效计算                    │
  │  │                                           │
  │  稳态阶段 (Steady): ceil(K/R) × ceil(N/C)   │
  │  │ 所有PE同时有效计算                        │
  │  │ 利用率 = 100%                             │
  │  │                                           │
  │  排空阶段 (Drain):  R+C-1 周期               │
  │    数据流过阵列, 部分PE已空闲                 │
  │                                             │
  │  总周期 = Fill + Steady + Drain              │
  │         = (R+C-1) + Steady + (R+C-1)         │
  │                                             │
  │  开销比 = 2×(R+C-1) / 总周期                 │
  └─────────────────────────────────────────────┘

本设计: R=16, C=16
  Fill+Drain = 2×(16+16-1) = 62 周期

对于大矩阵 (K=800, N=64):
  Steady ≈ ceil(800/16) × ceil(64/16) = 50×4 = 200 周期
  开销比 = 62 / (62+200) = 23.7% (单次计算)

但对于多tile的情况, 每个tile的Fill/Drain开销可以部分隐藏:
  实测平均开销 ≈ 4.7% (得益于tile间的流水线重叠)
```

### 3.3 瓶颈二：DMA数据搬运延迟

```
DMA搬运在计算之前发生, 造成计算单元空闲等待:

时间线:
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │DMA搬运│ │计算   │ │DMA搬运│ │计算   │
  │Layer1 │ │Layer1 │ │Layer2 │ │Layer2 │
  └──────┘ └──────┘ └──────┘ └──────┘
  ◄───串行执行, DMA期间计算单元空闲───┘

量化分析 (以Layer 2为例):
  DMA搬运: 5600 周期 (im2col变换 + 写入SA_RAM)
  MAC计算: 11872 周期
  DMA占比: 5600 / (5600+11872) = 32.1%

但DMA延迟并不完全浪费:
  - DMA搬运Layer N+1的权重时, 可以同时计算Layer N
  - 实际DMA延迟开销 ≈ 6.2% (通过流水线重叠部分隐藏)
```

**优化方向：DMA与计算重叠**

```
当前实现 (串行):
  DMA_L1 → MAC_L1 → DMA_L2 → MAC_L2 → DMA_L3 → MAC_L3
  ◄────────────────── 总延迟 ──────────────────────┘

优化后 (重叠):
  DMA_L1 → DMA_L2 → DMA_L3 ...
           MAC_L1 → MAC_L2 → MAC_L3 ...
  ◄──── 总延迟(由较长的链决定) ────┘

需要: 双缓冲(Double Buffering)
  Buffer A: 当前计算使用的数据
  Buffer B: DMA正在加载的下一组数据
  计算完成后切换A/B角色
```

### 3.4 瓶颈三：层间流水线气泡

```
层间切换时的额外开销:

┌─────────────────────────────────────────────┐
│  1. CSR配置时间                              │
│     CPU写NPU寄存器 (每层约5-10次AXI-Lite写)  │
│     每次AXI-Lite写: 3-4 周期                 │
│     总计: ~30-40 周期/层                      │
│                                             │
│  2. 流水线排空与重填                          │
│     脉动阵列需要排空上一层的残留数据           │
│     新一层需要重新填充                        │
│     开销: ~62 周期/层 (16+16-1的两倍)         │
│                                             │
│  3. 存储切换                                 │
│     im2col源切换 (image_buf → pool_buf)      │
│     权重Buffer切换                           │
│     开销: ~5-10 周期/层                       │
│                                             │
│  总层间开销: ~100-110 周期/层                 │
│  对于3层网络 (Conv1+Conv2+FC):               │
│  总气泡 = 2 × 100 ≈ 200 周期                 │
│  占总推理时间: 200/22715 ≈ 0.9% (TinyCIFAR)  │
│  对于ResNet-18 (18层): 18 × 100 = 1800周期   │
│  占比: 1800/821304 ≈ 0.2% (可忽略)           │
└─────────────────────────────────────────────┘

实测层间气泡开销 ≈ 6.4%
(包含更深层网络的累积效应和非计算层如BN/ReLU的配置)
```

## 4. 总线带宽利用率分析

### 4.1 理论带宽模型

```
AXI Burst传输的理论利用率:

η = L / (1 + L) × 100%

其中 L = Burst长度 (beat数)

┌──────────┬──────────────┐
│ Burst长度 │ 理论利用率    │
├──────────┼──────────────┤
│ 1        │ 50.0%        │
│ 4        │ 80.0%        │
│ 8        │ 88.9%        │
│ 16       │ 94.1%        │
│ 256      │ 99.6%        │
└──────────┴──────────────┘

本设计: DMA_MAX_BEAT_BURST = 16
理论最大利用率: 94.1%
```

### 4.2 实测带宽利用率

|----------|-----------|----------|----------|----------|--------|
| NPU读权重 (Bank1) | 16 | 4096 B | 544 | 1.506 GB/s | 88.9% |

**CPU利用率低的原因：**

```
CPU的Burst长度通常较短 (L=4):
  η = 4 / (1+4) = 80% (理论)
  实测: 67.1% (因额外的总线仲裁开销)

但CPU带宽仅占系统总带宽的3.2%, 对整体影响微乎其微
```

### 4.3 带宽利用率提升策略

**源码位置：** `docs/DESIGN_REPORT.md` 第3.3.8节

| 优化措施 | 实现方式 | 收益 |
|----------|----------|------|
| 统一32-bit数据位宽 | AXI_DATA_W=32，单拍4字节 | 简化设计，无需位宽转换 |
| Outstanding事务 | OSTDREQ_NUM=4，每Master 4深度 | 隐藏地址握手延迟 |
| 读写完全独立 | AR/AW、R/B通道分模块并行处理 | 读写互不阻塞 |

| Round-Robin无饥饿 | Mask向量机制确保公平 | 避免低优先级饿死 |
| 乱序完成支持 | per-ID FIFO + RR仲裁 | 最大化响应通道利用率 |

## 5. 优化方向

### 5.1 数据流优化

```
优化1: 双缓冲 (Double Buffering)
  ┌─────────────────────────────────────────────┐
  │  当前: DMA搬运 → 计算 → DMA搬运 → 计算       │
  │  优化: DMA写Buffer_B → 计算Buffer_A           │
  │         DMA写Buffer_A → 计算Buffer_B          │
  │         (搬运与计算完全重叠)                    │
  │                                             │
  │  预期收益: 消除DMA延迟瓶颈 (~6.2% → ~0%)     │
  │  代价: 存储空间翻倍                           │
  └─────────────────────────────────────────────┘

优化2: 流式im2col (已在本设计实现)
  ┌─────────────────────────────────────────────┐
  │  传统: 先完成全部im2col, 再开始MAC计算         │
  │  流式: im2col生成一行, MAC立即消费一行         │
  │                                             │
  │  本设计实现: dmac_im2col_stream (组合逻辑)    │
  │  im2col延迟: 0周期 (纯组合逻辑地址计算)       │
  └─────────────────────────────────────────────┘

优化3: 权重预取 (Weight Prefetching)
  ┌─────────────────────────────────────────────┐
  │  当前: 权重在计算开始前全部加载                 │
  │  优化: 边计算边加载下一tile的权重               │
  │                                             │
  │  需要: 权重Buffer双端口或双Bank                │
  │  预期收益: 减少权重加载开销 ~30%               │
  └─────────────────────────────────────────────┘
```

### 5.2 计算优化

```
优化4: Winograd变换 (针对3×3卷积)
  ┌─────────────────────────────────────────────┐
  │  标准卷积: 每个输出点需要 K×K = 9 次MAC       │
  │  Winograd: 每个输出点仅需 ~4 次MAC            │
  │                                             │
  │  加速比: 9/4 = 2.25× (理论)                  │
  │  实际: ~1.8× (考虑变换开销)                   │
  │                                             │
  │  适用: 3×3卷积 (ResNet主力)                   │
  │  不适用: 5×5卷积 (本设计网络)                  │
  └─────────────────────────────────────────────┘

优化5: 稀疏化加速 (Sparsity)
  ┌─────────────────────────────────────────────┐
  │  INT8量化后, 权重中存在大量零值                 │
  │  跳过零值计算可提升有效算力                     │
  │                                             │
  │  典型稀疏率: 30-60% (取决于网络和量化策略)      │
  │  等效算力提升: 1.4× ~ 2.5×                    │
  │                                             │
  │  实现: PE中增加零检测逻辑, 跳过MAC运算         │
  │  代价: 非规则数据流, 控制复杂度增加             │
  └─────────────────────────────────────────────┘

优化6: 混合精度 (Mixed Precision)
  ┌─────────────────────────────────────────────┐
  │  当前: 全INT8 (weight=8b, activation=8b)     │
  │  优化: 首末层用INT16, 中间层用INT8            │
  │                                             │
  │  首层对精度敏感 (输入噪声放大)                  │
  │  末层对精度敏感 (分类决策边界)                  │
  │  中间层容忍度高                                │
  │                                             │
  │  预期: 准确率提升1-2%, 算力损失<5%             │
  └─────────────────────────────────────────────┘
```

### 5.3 架构优化

```
优化7: 增加Core数量 (线性扩展)
  ┌─────────────────────────────────────────────┐
  │  当前: 8核 × 256 MAC = 2048 MAC → 0.82 TOPS  │
  │  扩展: 12核 × 256 MAC = 3072 MAC → 1.23 TOPS  │
  │                                             │
  │  面积代价: +50% (线性)                        │
  │  功耗代价: +50% (线性, 但可用CG/PG补偿)       │
  │  总线压力: 需要更宽的数据位宽或更高频率         │
  └─────────────────────────────────────────────┘

优化8: 增大阵列尺寸 (非线性扩展)
  ┌─────────────────────────────────────────────┐
  │  当前: 16×16 per Core                        │
  │  扩展: 32×32 per Core                        │
  │                                             │
  │  MAC数: 256 → 1024 (4×)                      │

  │  大矩阵利用率更高                              │
  │  小矩阵利用率可能下降 (浪费更多PE)             │
  └─────────────────────────────────────────────┘

优化9: 片上存储扩大
  ┌─────────────────────────────────────────────┐
  │  当前: ~328 KB (权重+特征图+中间结果)          │
  │  扩展: 512 KB 或 1 MB                        │
  │                                             │
  │  好处: 支持更大的网络 (如ResNet-50)            │
  │  减少片外访存 (如果未来接DDR)                  │
  │  代价: SRAM面积增大                           │
  └─────────────────────────────────────────────┘

## 8. 总结与展望

### 8.1 本设计的创新点

```
1. 瓦片化可扩展NPU架构
   · 4×4为基本瓦片, 16×16/Core, 8 Core并行
   · 从0.1 TOPS到1.6 TOPS的线性算力扩展
   · 兼顾基础要求与优化目标

2. 动态可调脉动阵列
   · Core Mask运行时配置活跃Core数量
   · 未使用Core自动门控 (CG/PG)
   · 算力与功耗动态平衡

3. 硬件原生层序列器
   · Conv/Pool/FC/Activation硬件执行
   · CPU仅需配置少量CSR
   · 大幅减少CPU干预开销

4. 分层低功耗体系
   · Core级时钟门控 (即时生效)
   · NPU级DFS (几周期切换)
   · Core级电源门控 (深度节能)
   · 全速→深度睡眠的宽范围调节

5. 标准化AXI接口
   · AXI-Lite CSR (CPU配置)
   · AXI4-Full Burst (数据搬运)
   · 与任何AXI兼容CPU可复用
```

### 8.2 不足与改进方向

```
当前不足:

1. 算力未达1 TOPS
   · 8核配置仅0.82 TOPS
   · 需扩展至12核才能突破1 TOPS

2. 不支持稀疏计算
   · INT8量化后存在大量零权重
   · 未实现零值跳过加速

3. 仅支持INT8精度
   · 不支持FP16/INT4等混合精度
   · 某些网络对精度敏感

4. FPGA验证未完成
   · 仅完成RTL仿真验证
   · FPGA综合资源占用约35%

5. 无片外存储接口
   · 仅支持片上SRAM (256KB)
   · 大型网络需要DDR支持
```

### 8.3 未来展望

```
短期 (6个月):
  · 完成FPGA原型验证 (Xilinx UltraScale+)
  · 支持更多网络 (MobileNet, EfficientNet-Lite)
  · 实现稀疏化加速

中期 (1年):
  · 扩展至12核, 突破1 TOPS
  · 支持混合精度 (INT8/INT16/FP16)
  · 增加DDR控制器, 支持大模型

长期 (2年):
  · ASIC流片 (28nm或14nm)
  · 集成RISC-V向量扩展
  · 支持训练 (反向传播)
  · 面向边缘AI产品化
```

## 11. 常见问题

**Q1: 为什么实测算力低于理论峰值？**

理论峰值假设每个周期每个MAC都在做有效计算。实际中存在：(1) 脉动阵列填充排空的无效周期；(2) DMA搬运期间计算单元空闲；(3) 层间切换的配置开销。本设计通过流式im2col和tile化计算，将这些开销控制在17%以内。

**Q2: 增加Core数量是否线性提升算力？**

理论上是的（TOPS ∝ MAC_count）。但实际中受限于：(1) 总线带宽——更多Core需要更多数据供给；(2) 存储容量——更多Core需要更大Buffer；(3) 面积功耗——FPGA资源有限。本设计在8核时总线利用率86.7%，若增至16核可能需要128-bit数据位宽。

**Q3: Winograd变换为什么不适合本设计？**

Winograd变换针对3×3卷积优化（2.25×加速），但本设计的TinyCIFAR10_5x5网络使用5×5卷积核。Winograd对5×5核的加速比有限（约1.4×），且变换本身的额外开销较大。对于以3×3为主的大网络（如ResNet），Winograd是值得考虑的优化。

**Q4: 代码覆盖率97.2%够不够？**

对于竞赛项目，97.2%已超过95%的要求。未覆盖的2.8%主要是：(1) 不可达的异常保护代码（如超时复位）；(2) 综合优化掉的冗余逻辑；(3) 极端边界条件（如Burst长度=0）。在商用项目中，通常要求行覆盖率>95%，分支覆盖率>90%，条件覆盖率>85%。
