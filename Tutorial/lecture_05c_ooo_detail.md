# Lecture 05c: AXI Crossbar深入（五）— 乱序完成机制详解

## 课程目标

本讲深入分析 `axicb_slv_ooo.sv`（336行）——Crossbar的乱序完成管理器，理解多Outstanding事务的跟踪和路由机制。

---

## 1. 模块概览

### 1.1 为什么需要乱序完成？

```text
场景：CPU连续发起两个读请求
  请求0: 读DDR (快，10周期返回)
  请求1: 读NPU CSR (慢，50周期返回)

如果严格按序完成：
  请求0发出 → 等请求0返回 → 请求1发出 → 等请求1返回
  总时间: 10 + 50 = 60周期

如果允许乱序：
  请求0发出 → 请求1发出 → 请求0返回(10周期) → 请求1返回(50周期)
  总时间: max(10, 50) = 50周期（节省17%）

但乱序带来问题：
  响应返回时，如何知道它对应哪个请求？
  → 需要跟踪机制
```

### 1.2 slv_ooo的角色

```text
┌─────────────────────────────────────────────────────────┐
│                    axicb_slv_ooo                        │
│                                                         │
│  功能: 跟踪每个Master发出的事务，将响应路由回正确的Master│
│                                                         │
│  输入: 各Slave的B/R响应（含ID和Slave索引）              │
│  输出: 路由回各Master的B/R响应                          │
│                                                         │
│  关键: 用per-ID FIFO记录每个事务的来源                  │
└─────────────────────────────────────────────────────────┘
```

## 2. 设计视角：为什么这样设计？

### 2.1 方案对比

```text
┌─────────────┬──────────────────┬──────────────┬─────────────┐
│ 方案        │ 原理             │ 优点         │ 缺点        │
├─────────────┼──────────────────┼──────────────┼─────────────┤
│ A.per-ID    │ 每个ID一个FIFO   │ 支持乱序     │ 面积较大    │
│ FIFO跟踪    │ 记录事务属性     │ 同ID保序     │             │
│ (本设计)    │                  │              │             │
├─────────────┼──────────────────┼──────────────┼─────────────┤
│ B.全局FIFO  │ 一个FIFO记录所有 │ 面积最小     │ 不支持乱序  │
│             │ 事务             │              │ 同ID也乱序  │
├─────────────┼──────────────────┼──────────────┼─────────────┤
│ C.无跟踪    │ Slave直接返回    │ 无面积开销   │ 无法路由    │
│             │                  │              │ 响应        │
└─────────────┴──────────────────┴──────────────┴─────────────┘

选择A的理由：
  1. AXI规范要求：同ID事务必须按序完成
  2. 不同ID事务可以乱序完成（提升带宽）
  3. per-ID FIFO天然保证同ID按序
```

### 2.2 设计约束

```text
┌──────────────────────────────────────────────────────────────┐
│ 约束类型     │ 具体约束                    │ 来源           │
├──────────────────────────────────────────────────────────────┤
│ 保序约束     │ 同ID事务必须按序完成         │ AXI规范        │
│ 乱序允许     │ 不同ID事务可以乱序完成       │ AXI规范        │
│ Outstanding  │ 每个Master最多4个事务        │ OSTDREQ_NUM=4  │
│ ID空间       │ 8位ID，每Master有唯一前缀   │ ID_MASK        │
│ 错误优先     │ Misroute事务优先处理         │ 可靠性要求     │
└──────────────────────────────────────────────────────────────┘
```

## 3. 设计视角：如何从零开始设计？

### Step 1: 事务捕获（Grab阶段）

```text
当Master发出AW/AR请求时，捕获事务属性：

  捕获的信息:
    - ID: 事务标识（用于路由回正确的Master）
    - ALEN: burst长度（用于计数数据拍数）
    - Slave Index: 目标Slave编号（用于匹配响应来源）
    - Misroute Flag: 是否路由错误

  存储位置: per-ID FIFO
    FIFO索引 = ID ^ MST_ID_MASK（去除Master前缀）
    每个FIFO深度 = OSTDREQ_NUM（本项目=4）
```

### Step 2: 响应匹配（Arbitrate阶段）

```text
当Slave返回B/R响应时，需要找到对应的事务记录：

  匹配条件:
    - Slave Index 与响应来源一致
    - ID 与记录中的ID一致

  仲裁:
    - 多个匹配时，选择最老的（FIFO顺序）
    - Misroute事务优先（错误处理优先）
```

### Step 3: 响应路由（Complete阶段）

```text
匹配成功后，将响应路由回正确的Master：

  从FIFO中提取:
    - Master ID → 决定路由回哪个Master
    - ALEN → 决定是否还有更多数据拍（R通道）

  驱动:
    - 将B/R信号连接到对应Master的slv_if
```

### Step 4: FIFO管理

```text
per-ID FIFO的操作:
  Push: AW/AR握手时，将事务属性推入FIFO
  Pop: B/R握手完成时，将事务属性从FIFO弹出

  同一ID的事务保证FIFO顺序（先入先出）
  不同ID的事务独立管理
```

### Step 5: 验证策略

```text
1. 保序验证: 同一ID的两个事务，第二个必须在第一个之后完成
2. 乱序验证: 不同ID的两个事务，可以乱序完成
3. Outstanding满: 4个事务未完成时，第5个事务被阻塞
4. Misroute: 路由错误的事务优先返回错误响应
```

## 4. 设计视角：架构模式

### 模式: Per-ID FIFO跟踪

```text
┌─────────────────────────────────────────────────────────────┐
│ 模式: Per-ID FIFO 事务跟踪                                  │
│                                                             │
│ 核心: 每个事务ID一个FIFO，记录事务属性                       │
│                                                             │
│ 实现:                                                        │
│   - FIFO索引 = ID (或ID ^ MASK)                             │
│   - Push时记录: {ALEN, Slave_Idx, Misroute, Master_ID}      │
│   - Pop时提取: 用于路由响应                                  │
│                                                             │
│ 特性:                                                        │
│   - 同ID保序: FIFO天然先入先出                               │
│   - 不同ID独立: 各FIFO互不影响                               │
│   - Outstanding限制: FIFO深度 = 最大事务数                   │
│                                                             │
│ 复用: 任何需要跟踪多个Outstanding事务的互连                  │
│ 例如: AXI Crossbar、PCIe switch、NoC路由器                   │
└─────────────────────────────────────────────────────────────┘
```

## 5. 三级流水线详解

### 5.1 Stage 1: Grab（捕获）

```text
触发条件: AW/AR握手 (awvalid && awready 或 arvalid && arready)

操作:
  1. 从awch/arch中提取 {ID, ALEN, 地址}
  2. 根据地址确定目标Slave Index
  3. 计算FIFO索引: idx = ID ^ MST_ID_MASK
  4. 将 {ALEN, Slave_Idx, Misroute} 推入FIFO[idx]
```

### 5.2 Stage 2: Arbitrate（仲裁）

```text
触发条件: 来自Slave的B/R响应 (bvalid 或 rvalid)

操作:
  1. 遍历所有per-ID FIFO，找到匹配的事务记录
  2. 匹配条件: FIFO中的Slave_Idx == 响应来源Slave
  3. 多个匹配时: 选择最老的（FIFO头部）
  4. Misroute优先: 如果有misroute事务，优先处理
```

### 5.3 Stage 3: Complete（完成）

```text
触发条件: 仲裁选出获胜事务

操作:
  1. 从FIFO中提取 Master_ID
  2. 将B/R信号路由回对应Master的slv_if
  3. 对于R通道: 用ALEN计数，最后一拍时Pop FIFO
  4. 对于B通道: 直接Pop FIFO
```

## 6. 本讲要点

| 要点 | 说明 |
|------|------|
| 乱序完成 | 不同ID事务可以乱序，提升带宽 |
| 同ID保序 | 同一ID的事务必须按序完成（FIFO保证） |
| per-ID FIFO | 每个ID一个FIFO，记录事务属性 |
| 三级流水 | Grab → Arbitrate → Complete |
| Misroute优先 | 路由错误的事务优先处理 |
| Outstanding限制 | FIFO深度 = 最大并发事务数 |
