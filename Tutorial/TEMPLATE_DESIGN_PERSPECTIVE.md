### 设计视角 1: 为什么这样设计？（WHY）

> 回答：这个模块要解决什么问题？有哪些约束？为什么选这个方案而不是其他？

#### 1. 设计动机

```text
问题：CPU需要配置DMA的寄存器（源地址、目的地址、传输字节数等），
      这些寄存器需要通过AXI总线被CPU访问。

约束：
  - 赛题强制要求：必须通过AXI-Lite memory-mapped方式访问（不能用PCPI自定义指令）
  - AXI-Lite是标准协议，通用性好，可复用于任何支持AXI的CPU
  - 寄存器数量有限（~20个），不需要burst传输
```

#### 2. 设计备选方案对比

```text
方案A: AXI-Lite CSR（本设计采用）
  优点: 标准协议、接口简单、面积小、CPU可直接访问
  缺点: 单拍传输，批量配置时效率低
  适用: 寄存器数量<64，访问频率不高

方案B: AXI4-Full CSR
  优点: 支持burst传输，批量配置效率高
  缺点: 接口复杂，面积大，寄存器场景不需要burst
  适用: 需要burst访问的场景（如配置大量查找表）

方案C: 自定义寄存器接口（非标准）
  优点: 最简单，面积最小
  缺点: 不通用，换CPU需要重新适配
  适用: 专用SoC，不需要通用性

选择A的理由：
  1. 赛题强制要求AXI-Lite
  2. DMA寄存器数量少（~20个），单拍传输足够
  3. AXI-Lite是业界标准，学习价值高
```

#### 3. 设计约束清单

```text
┌─────────────────────────────────────────────────────────┐
│ 约束类型        │ 具体约束                │ 来源        │
├─────────────────────────────────────────────────────────┤
│ 协议约束        │ 必须AXI-Lite            │ 赛题要求    │
│ 功能约束        │ 支持2个描述符            │ DMA规格     │
│ 功能约束        │ 支持go/abort控制         │ DMA规格     │
│ 接口约束        │ 32位数据宽度            │ AXI-Lite    │
│ 接口约束        │ 4字节地址对齐            │ AXI-Lite    │
│ 时序约束        │ 写响应3拍内返回          │ 性能要求    │
│ 时序约束        │ 读响应2拍内返回          │ 性能要求    │
│ 面积约束        │ 尽量小（CSR不是性能瓶颈） │ 系统权衡    │
└─────────────────────────────────────────────────────────┘
```

### 设计视角 2: 如何从零开始设计？（HOW）

> 回答：如果让你从空白开始设计这个模块，你会怎么思考？

#### Step 1: 明确接口

```text
设计的第一步：列出所有输入输出信号。

从需求出发：
  - 需要被CPU通过AXI-Lite访问 → 需要AW/W/B/AR/R五组信号
  - 需要输出控制信号给DMA FSM → go, abort, max_burst
  - 需要输入状态信号从DMA FSM → done, error
  - 需要输出描述符给DMA FSM → src_addr, dst_addr, num_bytes, cfg (×2)

接口设计原则：
  - AXI-Lite侧：遵循标准信号命名（i_awvalid, o_awready, ...）
  - 功能侧：使用有意义的信号名（o_dma_control_go, i_dma_status_done）
  - 参数化：ID_WIDTH, ADDR_WIDTH, DATA_WIDTH 可配置
```

#### Step 2: 设计寄存器映射

```text
设计的第二步：确定每个寄存器的地址、位域、读写属性。

寄存器设计原则：
  1. 地址4字节对齐（AXI-Lite要求）
  2. 控制寄存器（RW）和状态寄存器（RO）分开
  3. 触发信号（go, abort）设计为自清零（写1触发，硬件自动清零）
  4. 保留地址空间给未来扩展

设计过程：
  先列出所有需要的寄存器 → 分配地址 → 定义位域 → 确定读写属性
```

#### Step 3: 设计写通道状态机

```text
设计的第三步：AXI-Lite写操作需要处理AW和W通道的异步到达。

关键设计决策：AW和W谁先到？
  - AXI规范允许AW和W以任意顺序到达
  - 需要一个机制来"等待"两个通道都就绪后再执行写操作

方案：使用hold寄存器
  - AW先到：锁存地址到awaddr_q，置aw_hold=1，等待W
  - W先到：锁存数据到wdata_q，置w_hold=1，等待AW
  - 两者都到：执行写操作，清除hold标志，返回B响应

这是AXI-Lite从接口的标准设计模式，掌握后可复用于任何CSR模块。
```

#### Step 4: 设计读通道逻辑

```text
设计的第四步：读操作比写操作简单，因为只有AR和R两个通道。

关键设计决策：读数据什么时候有效？
  - 方案A：AR握手后1拍返回R数据（本设计采用）
  - 方案B：AR握手后立即返回R数据（组合逻辑，时序差）
  - 方案C：AR握手后N拍返回R数据（用于外部存储器）

本设计选择方案A：1拍延迟，平衡时序和性能。
```

#### Step 5: 验证策略

```text
设计的第五步：如何验证设计的正确性？

验证维度：
  1. 功能正确性：写入寄存器后读回，值是否一致？
  2. 协议合规性：valid/ready握手是否符合AXI规范？
  3. 边界条件：非对齐地址、无效地址、同时读写
  4. 错误处理：SLVERR是否正确返回？
```

### 设计视角 3: 架构模式与原则（PATTERN）

> 回答：这个设计体现了什么通用的架构模式？可以复用到哪些场景？

#### 模式 1: AW/W 解耦模式

```text
┌─────────────────────────────────────────────────────────┐
│ 模式名称: AW/W Hold 寄存器解耦                           │
│                                                         │
│ 适用场景: AXI-Lite从接口需要同时接受独立的AW和W通道       │
│                                                         │
│ 核心思想: 用两个hold寄存器分别暂存先到达的通道数据，       │
│           等两个通道都就绪后再执行写操作。                  │
│                                                         │
│ 状态机:                                                  │
│   IDLE → (AW先到) → HOLD_AW → (W到) → EXEC → IDLE       │
│   IDLE → (W先到)  → HOLD_W  → (AW到)→ EXEC → IDLE       │
│   IDLE → (同时到) → EXEC → IDLE                          │
│                                                         │
│ 复用场景: 任何AXI-Lite从接口的寄存器文件                   │
│ 例如: NPU CSR、外设配置寄存器、中断控制器                  │
└─────────────────────────────────────────────────────────┘
```

#### 模式 2: 地址解码 + 多路选择模式

```text
┌─────────────────────────────────────────────────────────┐
│ 模式名称: 地址解码 + 多路选择                             │
│                                                         │
│ 适用场景: 多个寄存器通过统一地址空间访问                   │
│                                                         │
│ 核心思想: 地址的低位作为索引，选择目标寄存器。              │
│                                                         │
│ 实现:                                                    │
│   addr[7:2] → 6-bit索引 → 64个寄存器的选择信号            │
│   写操作: 用case语句根据索引选择写入目标                   │
│   读操作: 用case语句根据索引选择读出源                     │
│                                                         │
│ 设计要点:                                                │
│   - 地址对齐: AXI-Lite要求4字节对齐，addr[1:0]=00         │
│   - 用addr[7:2]而非addr[7:0]作为索引，节省解码逻辑        │
│   - 未命中地址返回SLVERR                                  │
│                                                         │
│ 复用场景: 任何基于地址映射的寄存器文件                      │
└─────────────────────────────────────────────────────────┘
```

#### 模式 3: 自清零触发信号模式

```text
┌─────────────────────────────────────────────────────────┐
│ 模式名称: 自清零触发信号                                  │
│                                                         │
│ 适用场景: 控制寄存器中的"触发"位（如go、abort）            │
│                                                         │
│ 问题: CPU写go=1启动DMA，但如果go保持为1，DMA会反复启动。   │
│                                                         │
│ 解决: 硬件在检测到go=1后，自动将其清零。                   │
│                                                         │
│ 实现:                                                    │
│   always_ff @(posedge clk) begin                        │
│     if (wr_en && addr==CONTROL && wdata[0])             │
│       go_reg <= 1'b1;       // CPU写1，置位              │
│     else if (go_reg)        // 下一周期                  │
│       go_reg <= 1'b0;       // 自动清零                  │
│   end                                                    │
│                                                         │
│ 效果: go信号只持续1个时钟周期，产生单周期脉冲。             │
│                                                         │
│ 复用场景: 任何需要"触发一次"的控制信号                      │
│ 例如: 中断清除、DMA启动、NPU启动、看狗喂狗                 │
└─────────────────────────────────────────────────────────┘
```

## 使用建议

1. **不需要重写所有讲**：先在重点章节（DMA、NPU、Crossbar）补充设计视角
2. **每讲只需补充2-3页**：设计视角不需要很长，关键是点出设计决策和权衡
3. **用对比表格**：备选方案对比用表格最清晰
4. **用流程图**：设计过程用Step 1/2/3的流程最易懂
5. **增加设计类练习**：让学习者自己做设计决策，而不只是看懂代码
