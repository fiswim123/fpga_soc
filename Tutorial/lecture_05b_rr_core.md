# Lecture 05b: AXI Crossbar深入（四）— Round-Robin仲裁核心

## 课程目标

本讲深入分析 `axicb_round_robin_core.sv`（2779行）——Crossbar中最复杂的模块，理解无饥饿轮询仲裁算法的实现。

---

## 1. 模块概览

### 1.1 仲裁器的角色

```text
场景：CPU和DMA同时访问DDR
  CPU:  awvalid[0] = 1, awaddr = 0x4000_0000
  DMA:  awvalid[1] = 1, awaddr = 0x4000_0100

DDR只有一个接口，必须选择一个 → 仲裁器

仲裁器的职责：
  输入: N个请求信号 req[N-1:0]
  输出: N个授权信号 grant[N-1:0] (one-hot)
  约束: 保证公平（无饥饿）、无死锁
```

### 1.2 参数化设计

```text
axicb_round_robin_core 支持 2~22 个请求者：
  - 通过 REQ_NB 参数配置
  - 使用 generate 块生成不同宽度的逻辑
  - 本项目: REQ_NB = 4 (4个Master)
```

---

## 2. 设计视角：为什么这样设计？

### 2.1 为什么用Mask-Based Round-Robin？

```text
方案A: 简单轮询（计数器）
  用计数器 0→1→2→3→0→1→... 依次服务
  问题: 如果req[0]一直有请求，req[3]要等3轮才能被服务
        计数器必须遍历所有位置，效率低

方案B: 固定优先级
  req[0] > req[1] > req[2] > req[3]
  问题: req[3]可能永远得不到服务（饥饿）

方案C: Mask-Based Round-Robin（本设计）
  用mask记录"上一次谁被服务"，下一次从它的下一个开始
  优点: 无饥饿、响应快、可参数化
  缺点: 实现稍复杂

选择C的理由：
  1. 无饥饿：每个请求者最多等N-1轮
  2. 响应快：不需要遍历所有位置
  3. 可参数化：2~22个请求者用同一套算法
```

### 2.2 设计约束

```text
┌──────────────────────────────────────────────────────────────┐
│ 约束类型     │ 具体约束                    │ 来源           │
├──────────────────────────────────────────────────────────────┤
│ 公平性       │ 每个请求者必须被服务         │ 无饥饿要求     │
│ 响应性       │ 单个请求者时立即授权         │ 性能要求       │
│ 无死锁       │ 任何请求组合都能产生授权     │ 可靠性要求     │
│ 参数化       │ 支持2~22个请求者             │ 通用性要求     │
│ 面积         │ 尽量小（每个Slave一个实例）  │ 4个实例        │
└──────────────────────────────────────────────────────────────┘
```

## 3. 设计视角：如何从零开始设计？

### Step 1: 理解Mask机制

```text
Mask是一个N位向量，记录"允许被服务的范围"。

初始状态: mask = 4'b1111 (所有人都允许)

第1轮: req = 4'b1010 (req[1]和req[3]有请求)
  masked = mask & req = 4'b1010
  从低位找第一个1 → grant[1]
  更新mask: 将grant[1]位清零，更低位置1
  新mask = 4'b1100 (下次从req[2]开始找)

第2轮: req = 4'b1010
  masked = 4'b1100 & 4'b1010 = 4'b1000
  从低位找第一个1 → grant[3]
  更新mask = 4'b1111 (到了最高位，重新开始)

第3轮: req = 4'b1010
  masked = 4'b1111 & 4'b1010 = 4'b1010
  从低位找第一个1 → grant[1]
  ...

结果: req[1]和req[3]轮流被服务，公平！
```

### Step 2: 实现"从低位找第一个1"

```text
这是仲裁器的核心操作：Priority Encoder。

  输入: masked[N-1:0]
  输出: grant[N-1:0] (one-hot)

实现方式：
  方式A: for循环遍历
    for (i = 0; i < N; i++)
      if (masked[i]) begin grant[i] = 1; break; end

  方式B: 递归生成（本设计）
    将N位拆成两半，递归找第一个1
    面积更优，时序更好

  方式C: 并行前缀树
    最快，但面积最大
```

### Step 3: 处理"孤独请求"

```text
问题：如果mask范围内没有活跃请求怎么办？

  mask = 4'b1100 (只允许req[2]和req[3])
  req  = 4'b0011 (只有req[0]和req[1]有请求)
  masked = 4'b1100 & 4'b0011 = 4'b0000 (空！)

解决：回退到全req空间查找
  如果masked == 0，则在原始req中找第一个1
  这确保了即使mask限制了范围，也不会漏掉请求
```

### Step 4: 更新Mask

```text
授权grant[i]后，更新mask：
  新mask的位j = 1 当且仅当 j > i

  例: grant[2] → mask = 4'b1000 (只允许req[3])
      grant[3] → mask = 4'b1111 (最高位，重新开始)
      grant[0] → mask = 4'b1110 (允许req[1],req[2],req[3])

实现：
  new_mask[j] = (j > granted_index) ? 1 : 0
```

### Step 5: 验证策略

```text
1. 公平性: 连续100轮仲裁，统计每个请求者被服务的次数
2. 无饥饿: 一个请求者连续1000次请求，另一个偶尔请求，后者不被饿死
3. 单请求: 只有一个请求者时，每轮都被授权（无额外延迟）
4. 全请求: 所有请求者同时请求时，轮流授权
```

## 4. 设计视角：架构模式

### 模式: Mask-Based Round-Robin

```text
┌─────────────────────────────────────────────────────────────┐
│ 模式: Mask-Based Round-Robin 仲裁                           │
│                                                             │
│ 核心: 用mask记录"上一次服务到谁"，从下一个开始找             │
│                                                             │
│ 算法:                                                        │
│   1. masked = mask & req     // 只看允许范围内的请求         │
│   2. grant = first_one(masked) // 找第一个1                  │
│   3. if (masked == 0)                                         │
│        grant = first_one(req)  // 回退到全范围               │
│   4. mask = update(grant)    // 更新mask                     │
│                                                             │
│ 特性:                                                        │
│   - 无饥饿: 每个请求者最多等N-1轮                            │
│   - 无死锁: 任何请求组合都能产生grant                        │
│   - 响应快: 单请求时0额外延迟                                │
│                                                             │
│ 复用: 任何需要公平仲裁的场景                                 │
│ 例如: 总线仲裁、内存调度、网络交换机                         │
└─────────────────────────────────────────────────────────────┘
```

## 5. 算法追踪示例

### 5.1 四请求者追踪

```text
时间   req      mask     masked   grant   新mask
────   ────     ────     ──────   ─────   ──────
T0     1011     1111     1011     0001    1110
T1     1010     1110     1010     0010    1100
T2     1000     1100     1000     1000    1111
T3     1011     1111     1011     0001    1110
T4     1010     1110     1010     0010    1100
T5     1000     1100     1000     1000    1111

观察：
  - req[0]在T0,T3被服务（间隔2轮）
  - req[1]在T1,T4被服务（间隔2轮）
  - req[3]在T2,T5被服务（间隔2轮）
  - 公平！每个请求者轮流被服务
```

### 5.2 孤独请求追踪

```text
时间   req      mask     masked   grant   新mask
────   ────     ────     ──────   ─────   ──────
T0     0100     1111     0100     0100    1100
T1     0001     1100     0000     → 回退: 0001    1110

T1时: mask=1100(只允许req[2],req[3])，但req只有req[0]
     masked=0000(空) → 回退到req中找 → grant[0]
     新mask=1110(从req[1]开始)
```

## 6. 本讲要点

| 要点 | 说明 |
|------|------|
| Mask-Based RR | 用mask记录上次服务位置，从下一个开始 |
| 无饥饿 | 每个请求者最多等N-1轮 |
| 孤独请求 | mask范围内无请求时回退到全范围 |
| 参数化 | 支持2~22个请求者 |
| 优先级编码 | 从低位找第一个1（first_one操作） |
